Built for speed

Three pieces, all pointed at throughput.

Records for state, a pipeline that is built once instead of per call, and a fluent interface that only lets you build the fast, correct thing. None of it needs a broker, a database, or a config file.

Records

State with no locks

All pipeline state is one immutable record; each step returns a new one with with. No shared mutable state means no locks and no defensive copying, so parallel branches never contend. The record is also the design: non-null fields are inputs, nullable fields are outputs the steps fill in.

Compiled pipelines

Resolved once, not per call

You build the pipeline once at startup. EvalApp folds the whole tree, steps, gates, branches, sagas, into a single nested delegate. Every run after that is a chain of direct calls with no reflection, no container lookup, and no middleware re-assembly. On the plain path a step hands back its ValueTask with no async state machine at all.

Guided fluent interface

Correct concurrency you don't hand-write

Bounded parallelism, saga rollback, exception aggregation are each solved but hard to combine under pressure. The type-state builder offers only the moves valid where you are, and refuses to compile the invalid ones, after .BeginSaga() it requires .EndSaga(). You get fast, safe concurrency without writing the fiddly parts.

Throughput

It finds the fastest concurrency for your hardware.

Declare parallelism with a ForEach or a resource gate, and EvalApp finds the concurrency your hardware rewards by measuring real wall-time on every run. In integration testing that has meant 30–40% more throughput than hand-tuned baselines, from the same hardware, with no thread-count constant to guess and none to go stale when the environment shifts.

Adaptive · hill-climbing

Probe, measure, keep what's faster

A four-phase tuner probes the concurrency up and down, keeps a change only when it beats the recent best by a real margin, converges, then keeps watching for regressions and backs off if throughput drops.

Bayesian · Thompson sampling

Warm-starts after a restart

An opt-in variant treats each concurrency level as a bandit arm with its own posterior. The posteriors survive process restarts, so a freshly deployed instance starts near last time's best instead of re-exploring from scratch.

Declared parallelism · C# 
.ForEach(
    name:      "ProcessOrders",
    select:    d => d.Orders,
    configure: inner => inner
        .Gate(ResourceKind.Database, g => g
            .AddStep("Save", async (o, ct) => o with { Saved = await db.Save(o, ct) })),
    merge:     (orig, results) => orig with { Orders = results })

Below a small-collection threshold it runs the items inline, because tiny batches don't earn their scheduling cost. The tuner learns the concurrency your hardware rewards by measuring throughput, rather than a number you hard-code and hope stays right.

60-second start

Build once at startup. Call on every request.

Define the record, chain the steps, build. The same pipeline is thread-safe, reusable, and self-tuning from the first call.

C# · .NET 8 
// 1. One immutable record holds the entire pipeline state
record OrderData(int OrderId, bool IsValid = false, decimal Price = 0, string? Receipt = null);

// 2. Build once — the topology is resolved into a pre-built delegate here
Eval.App("ProcessOrder")
    .WithTuning()
    .DefineDomain("Orders")
        .DefineTask<OrderData>("Process")
            .AddStep("Validate",    d => d with { IsValid = d.OrderId > 0 })
            .AddStep("FetchPrice",  async (d, ct) => d with { Price = await catalog.GetPrice(d.OrderId, ct) })
            .AddStep("SendReceipt", async (d, ct) => d with { Receipt = await email.Send(d, ct) })
            .Run(out var pipeline)
        .Build();

// 3. Call on every request — reused, no per-call rebuild
var result = await pipeline.RunAsync(new OrderData(orderId));
switch (result)
{
    case PipelineResult<OrderData>.Success s: await Save(s.GetData()); break;
    case PipelineResult<OrderData>.Failure f: log.Error(f.Exception);  break;
    case PipelineResult<OrderData>.Skipped sk: log.Debug(sk.Reason);   break;
}

Add .BeginSaga() for steps that register compensation and roll back in reverse on failure, .If()/.Else() for conditional branches, and .Gate(ResourceKind.Network, ...) to bound and tune an I/O region, all in the same chain, all resolved into the one delegate at build time.

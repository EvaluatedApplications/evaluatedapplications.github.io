EvalApp — a pipeline engine that tunes itself to your hardware.

You declare what work to do, as a chain of small steps over one immutable data record. EvalApp works out how fast the machine it is running on can do it, and re-adapts when load or hardware changes. Zero dependencies, in-process, one DLL.

What it is

Declared parallelism, tuned at runtime.

Most .NET code defaults to sequential await chains because getting real parallelism right, correct throttling, exception aggregation, cancellation across every branch, is hard to sustain. EvalApp makes that a one-line declaration and then finds the concurrency your hardware actually rewards.

Self-tuning

Finds its own concurrency

The tuner watches real throughput on every run, probes neighbouring concurrency values, and converges on the number that suits the hardware, from the first run, with no benchmarking exercise.

→ 30–40% over hand-tuned baselines
Zero infrastructure

One DLL, no broker

Pure .NET 8 with no NuGet dependencies, no message broker, no external store, no config files. Add the package and it runs in-process. The tuner persists to a local JSON file on its own.

Compile-time safe

Whole bug classes can't be written

The type-state builder only offers the steps that are valid where you are. After .BeginSaga() the compiler requires .EndSaga() before it will run. Bad async structures simply don't compile.

→ the pipeline is its own architecture map
The idea

State is one record. Steps only touch the record.

The data record is the function signature and the shared state at once. Non-nullable fields are inputs, required to start. Nullable fields are outputs, null until the step that fills them runs. Adding a field touches only the steps that use it, so the refactoring cascade never happens.

The whole shape of a process, in one type 
record BatchData(
    int BatchId,                       // INPUT:   what batch to run
    DateTime ScheduledAt,              // INPUT:   when it was scheduled
    List<Item>?   LoadedItems,          // STAGE 1: loaded from the database
    List<Item>?   ValidatedItems,       // STAGE 2: passed validation
    List<string>? ProcessedIds,         // STAGE 3: processed successfully
    string?       ConfirmationRef       // STAGE 4: external confirmation
);

Design the record well and the steps almost write themselves. Reading the builder chain tells you, on one screen, which resources exist and at what limits, which steps are gated, which collections run in parallel, and which steps have rollback registered. It is always correct because it is the code.

60-second start

Build once at startup. Call on every request.

Define the record, chain the steps, and run. The same pipeline is thread-safe, reusable, and self-tuning from the first call.

C# · .NET 8 
// 1. One immutable record holds the entire pipeline state
record OrderData(int OrderId, bool IsValid = false, decimal Price = 0, string? Receipt = null);

// 2. Build once
EvalApp.App("ProcessOrder")
    .WithTuning()
    .DefineDomain("Orders")
        .DefineTask<OrderData>("Process")
            .AddStep("Validate",    d => d with { IsValid = d.OrderId > 0 })
            .AddStep("FetchPrice",  async (d, ct) => d with { Price = await catalog.GetPrice(d.OrderId, ct) })
            .AddStep("SendReceipt", async (d, ct) => d with { Receipt = await email.Send(d, ct) })
            .Run(out var pipeline)
        .Build(licenseKey: "YOUR-KEY");   // omit for the free sequential tier

// 3. Call on every request
var result = await pipeline.RunAsync(new OrderData(orderId));
switch (result)
{
    case PipelineResult<OrderData>.Success s: await Save(s.Data); break;
    case PipelineResult<OrderData>.Failure f: log.Error(f.Exception); break;
}

Parallel fan-out is a declaration too: .ForEach(...) runs the items concurrently in licensed mode and the tuner sets the degree for you. Unlicensed, the same call runs sequentially, same results, no tuner.

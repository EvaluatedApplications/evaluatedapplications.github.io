More than a pathfinder

The systems a game usually stitches together, in one place.

Most projects wire up four or five libraries for movement, crowds, combat and vision. NavPathfinder ships them as one coherent, headless engine over a single baked navmesh.

Move

Crowds that scale

Navmesh baking, single-agent A*, and flow-field crowd pathfinding where agents heading to the same goal share one field, so high agent counts stay cheap. Dynamic obstacles and multi-layer ground, air and water are built in.

Fight

Combat, not just collision

Resolve two forces in a tick: hit, dodge, block and crit, with damage applied and per-attack outcomes returned for your animation and audio. Bring your own RPG math through IGameMath; layer masks keep melee off flyers; a per-tick attack budget keeps it bounded.

Perceive

Vision and knowledge

Explored-persists fog of war keyed to navmesh line of sight, cone vision for "can this guard see the player", and influence maps that turn enemy, loot and objective positions into a per-triangle field your AI can sample.

AI movement

Behaviors, not a hand-written state machine.

Compose movement from scored behaviors. Each tick every behavior scores every agent, the highest wins, then it pathfinds. Guards patrol, civilians flee, crowds queue and lane, and you can add your own.

Built-in behaviors

Seek · Flee · Patrol · Wander · Queue · FlowLane

Drop-in behaviors, all real: flee a threat, cycle patrol waypoints, form an orderly queue at a chokepoint, or take the right-hand lane to break a head-on deadlock in a corridor. Add your own by implementing one interface.

Crowd smarts

Congestion, clustering, formations

Congestion analysis finds chokepoints and opposing traffic to drive queueing and laning. Under load, agents with nearby goals cluster so only representatives pathfind and the rest follow in formation, with a narrow-corridor guard so followers never get pushed into walls.

Quickstart

One world, the services you need, a tick loop.

Bake a navmesh once, then each frame feed agent snapshots and apply the waypoints. Everything hangs off a single PathfindingWorld.

C# · .NET 8 
// One world; pull the services you need. Add a license key for parallel + tuning.
var world   = new PathfindingWorld();
var baker   = world.CreateNavMeshBakerService();
var agents  = world.CreateMultiAgentService();

var navMesh = await baker.BakeAsync(vertices, triangles);

// Each frame: feed agent snapshots, get waypoints back.
var result = await agents.TickAsync(units, navMesh, tick);
foreach (var path in result.Paths)
    if (path.PathFound)
        Move(path.AgentId, path.Waypoints);

The same world vends the combat (CreateEngagementService), vision (CreateLineOfSightService, CreateFogOfWarService), influence-map, congestion, navigator and multi-population simulation services, and registers through services.AddNavPathfinder(...) for dependency injection.

Performance

Held to a frame budget, not a guessed constant.

There is no "max agents" number to hand-tune. NavPathfinder self-tunes its concurrency to the machine it runs on, and can manage its own work to a target frame time, reporting how much headroom is left.

Self-tuning

Finds the machine's limit

Built on our EvalApp engine, it measures real work per tick and converges on the concurrency the hardware rewards, then re-adapts when load changes.

Frame-budget modes

Manages to a target

Tuned mode keeps the tick under a target frame time by adapting how much new work it takes on, and reports back a pressure signal you can drive difficulty or spawn caps from.

Many populations

Split the budget

The simulation service runs several populations in one tick and splits the frame budget between them by weight, "units get 60%, civilians 40%", adjustable per tick.

~680 agents pathfinding and fighting inside an ~80 fps budget under full load, measured in a headless siege harness.
